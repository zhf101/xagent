# 审批流程阅读顺序清单

这份清单不是重复讲设计方案，而是给代码阅读提供一条稳定路径。

如果你要快速搞清楚“SQL 审批是怎么把一次工具调用变成可恢复工作流状态的”，按下面顺序读即可。

相关设计背景先看：

- `docs/sql审批流程设计.md`

---

## 1. 最短主链路

如果只想先读一遍主流程，按这 9 个文件走：

1. `src/xagent/web/models/task.py`
   先建立宿主状态概念。
   这里定义了 `TaskStatus.WAITING_APPROVAL`、`DAGExecutionPhase.WAITING_APPROVAL`，以及 `Task` / `DAGExecution` 上和审批阻断、恢复锚点有关的字段。

2. `src/xagent/web/models/sql_approval.py`
   再看审批专属账本模型。
   这里有 `ApprovalRequest` 和 `DAGStepRun`，分别代表“审批请求真相源”和“步骤执行事实表”。

3. `src/xagent/core/policy/sql_risk_analyzer.py`
   这里开始进入审批判定入口。
   先看 SQL 如何被规整、打风险等级、生成可复用指纹。

4. `src/xagent/core/policy/sql_policy_gateway.py`
   这是审批三段式 gate。
   重点看 `evaluate(...)`，理解它如何把风险分析、审批复用、待审批创建三步串起来。

5. `src/xagent/core/tools/core/sql_tool.py`
   这里看 SQL 真正执行前如何接入策略网关。
   重点不是数据库细节，而是“什么情况下直接执行，什么情况下返回 approval_required / policy_denied”。

6. `src/xagent/core/tools/adapters/vibe/sql_tool.py`
   这里看平台桥接层。
   它负责把 Task / Plan / Step 的 runtime 上下文注入 SQL 审批链路，并把工具结果落成 `DAGStepRun`。

7. `src/xagent/core/agent/pattern/react.py`
   这里看 ReAct 如何把工具层阻断结果原样带出。
   它不负责恢复，只负责把 `approval_required` 往上抛。

8. `src/xagent/core/agent/pattern/dag_plan_execute/plan_executor.py`
   这里是审批从“工具阻断”升级成“DAG 阻断”的关键点。
   重点看：
   - `_build_sql_policy_runtime_context(...)`
   - `_execute_step_with_react_agent(...)`
   - `_is_waiting_approval_react_result(...)`
   - `_is_waiting_approval_result(...)`

9. `src/xagent/core/agent/pattern/dag_plan_execute/dag_plan_execute.py`
   这里看宿主级 waiting_approval 状态如何形成并落快照。
   重点看：
   - `register_waiting_approval(...)`
   - `_persist_waiting_approval_snapshot(...)`

读完这 9 个文件，你就能串起下面这条主链：

`SQL tool call`
-> `risk analyze`
-> `policy decision`
-> `approval_required`
-> `step waiting_approval`
-> `DAG waiting_approval`
-> `Task/DAGExecution snapshot persisted`

## 2. 从执行到审批的代码跳转顺序

如果你希望按“函数调用链”阅读，而不是按模块层次阅读，推荐下面顺序：

1. `src/xagent/core/agent/pattern/dag_plan_execute/plan_executor.py`
   从 `_execute_step_with_react_agent(...)` 开始。
   这是 DAG step 调 ReAct 的入口。

2. `src/xagent/core/agent/pattern/dag_plan_execute/plan_executor.py`
   看 `_build_sql_policy_runtime_context(...)`。
   这里把 `task_id`、`plan_id`、`step_id`、`resume_token`、`dag_snapshot_version` 等恢复锚点组装好。

3. `src/xagent/core/tools/adapters/vibe/sql_tool.py`
   看 `execute_sql_query(...)` 和 `_execute_sql_query_with_policy(...)`。
   这里把运行时上下文交给审批网关，并负责生成 `DAGStepRun`。

4. `src/xagent/core/policy/sql_policy_gateway.py`
   看 `evaluate(...)`。
   这里决定是：
   - `allow_direct`
   - `wait_approval`
   - `deny`

5. `src/xagent/web/services/sql_approval_service.py`
   看 `create_or_reuse_pending_request(...)`。
   这里负责创建或复用待审批请求，并把审批请求和 resume token 绑定。

6. `src/xagent/core/tools/core/sql_tool.py`
   回来看执行结果如何转换成 `approval_required`。
   这是工具协议层的阻断结果。

7. `src/xagent/core/agent/pattern/react.py`
   看 `approval_required` 如何被保留并返回给上层。

8. `src/xagent/core/agent/pattern/dag_plan_execute/plan_executor.py`
   回来看 `_is_waiting_approval_react_result(...)` 分支。
   这里把 ReAct 结果翻译成 step 级 `waiting_approval`。

9. `src/xagent/core/agent/pattern/dag_plan_execute/dag_plan_execute.py`
   看 `register_waiting_approval(...)` 和 `_persist_waiting_approval_snapshot(...)`。
   这里把内存态阻断升级为宿主快照。

10. `src/xagent/web/api/websocket.py`
    看 `result.get("status") == "waiting_approval"` 分支。
    这里把运行结果第一次投影到 `Task`、聊天消息、trace 和前端广播。

## 3. 审批恢复链路阅读顺序

如果你关心的是“批准后如何继续跑”，按下面顺序读：

1. `src/xagent/web/api/sql_approval.py`
   从“批准审批请求”和“恢复任务”两个接口开始。
   先理解 web 层开放给页面的业务入口。

2. `src/xagent/web/services/sql_approval_service.py`
   重点看：
   - `approve_request(...)`
   - `reject_request(...)`
   - `get_request_by_resume_token(...)`
   - `expire_pending_requests(...)`

3. `src/xagent/web/services/dag_recovery_service.py`
   这是审批恢复编排核心。
   重点看：
   - `build_approval_summary(...)`
   - `can_resume(...)`
   - `resume_after_approval(...)`
   - `sync_expired_request_state(...)`

4. `src/xagent/core/agent/service.py`
   重点看 `_ensure_waiting_approval_can_resume(...)`。
   这是 continuation 场景下的兜底防线，防止绕开 web API 直接续跑。

5. `src/xagent/core/agent/pattern/dag_plan_execute/dag_plan_execute.py`
   看 pattern 恢复后如何继续从阻断 step 往下执行。

恢复主链可以概括成：

`approve request`
-> `load task + dag snapshot + approval request`
-> `validate resumable`
-> `AgentService secondary guard`
-> `resume blocked DAG`
-> `continue original step`

## 4. 按职责分层阅读

如果你不是按调用链读，而是想按“谁负责什么”来分层理解，可以这样分：

### 4.1 状态模型层

- `src/xagent/web/models/task.py`
- `src/xagent/web/models/sql_approval.py`

这一层回答的问题是：

- 平台里有哪些正式状态表达 `waiting_approval`
- 哪些字段是跨页面恢复锚点
- 哪些表是业务真相源，哪些只是运行态投影

### 4.2 审批策略层

- `src/xagent/core/policy/sql_risk_analyzer.py`
- `src/xagent/core/policy/sql_policy_gateway.py`

这一层回答的问题是：

- SQL 风险怎么判
- 为什么某条 SQL 要审批
- 已批准 SQL 为什么可以复用

### 4.3 工具桥接层

- `src/xagent/core/tools/core/sql_tool.py`
- `src/xagent/core/tools/adapters/vibe/sql_tool.py`

这一层回答的问题是：

- SQL 执行前在哪个边界接入审批
- 工具为什么不直接改 Task 状态
- `DAGStepRun` 为什么在这里落库

### 4.4 执行器状态机层

- `src/xagent/core/agent/pattern/react.py`
- `src/xagent/core/agent/pattern/dag_plan_execute/plan_executor.py`
- `src/xagent/core/agent/pattern/dag_plan_execute/dag_plan_execute.py`

这一层回答的问题是：

- 工具阻断如何变成 step 阻断
- step 阻断如何变成 DAG 阻断
- DAG 阻断如何沉淀成可恢复宿主快照

### 4.5 Web 宿主编排层

- `src/xagent/web/services/sql_approval_service.py`
- `src/xagent/web/services/dag_recovery_service.py`
- `src/xagent/web/api/sql_approval.py`
- `src/xagent/web/api/websocket.py`
- `src/xagent/core/agent/service.py`

这一层回答的问题是：

- 前端页面从哪里拿审批摘要
- 批准后为什么不是直接解锁，而是要走恢复编排
- 为什么 continuation 还要再校验一次审批状态

## 6. 建议的阅读目标

读这条链路时，建议带着下面 6 个问题去看：

1. 哪一层第一次发现“这条 SQL 不能直接执行”？
2. 哪一层把“工具阻断”翻译成 `waiting_approval` 状态机？
3. 哪一层负责把审批请求和 DAG 恢复锚点绑定起来？
4. 哪一层负责把内存态阻断变成跨页面可恢复状态？
5. 哪一层负责批准后的恢复编排？
6. 哪一层负责兜底，防止未批准直接续跑？

如果这 6 个问题你都能从代码里回答出来，审批主链路就已经读通了。

---

## 7. 一句话总览

可以把当前实现理解为：

“SQL 风险判断发生在工具执行前，审批请求真相源在 web 持久化层，`waiting_approval` 是 Task/DAG 的运行态投影，恢复靠 `ApprovalRequest + DAGExecution + Task` 三者组合完成，而不是依赖原进程内存继续活着。”
