# 智能造数平台基于 xagent 的改造计划 v1

## 1. 文档目标

本文用于汇总当前对 `xagent` 与“智能造数平台架构设计 v7”的对照分析，形成一版可继续细化的改造计划。

本文重点回答四个问题：

1. 这个需求是否适合直接基于 xagent 改造
2. 改造设计应该从 `ch5-base-agent-model` 还是 `ch8-react-pattern` 开始
3. 你的五层架构与 xagent 当前代码分别如何对应
4. 需要参考什么、保留什么、重做什么、分几阶段做

---

## 2. 总体结论

### 2.1 是否可以基于 xagent 改造

可以，但不是“换个 prompt + 配几个工具”就能完成。

更准确地说，xagent 已具备以下底座能力：

- `Agent / Pattern / Runner / Service` 的执行框架
- 现成的 `ReActPattern`
- 现成的 `DAGPlanExecutePattern`
- `MemoryStore`、LanceDB、历史恢复、trace、WebSocket
- SQL、HTTP API、文件、浏览器、MCP 等工具体系

但你的设计不是普通工作流，而是强调：

- `Agent` 是唯一业务决策源
- `Guard / Runtime / Resource / Ledger` 明确分层
- 所有非终止结果都要统一回流为 Observation
- `Human in Loop` 是监督通道，不是新的主脑

因此，改造目标不是“做一个新的领域 prompt”，而是：

**在 xagent 上新增一套造数领域控制内核。**

### 2.2 从 5 还是从 8 开始

建议采用下面的原则：

```text
阅读与边界设计：先看 ch5-base-agent-model
主循环改造与代码实现：从 ch8-react-pattern 开始
```

原因：

- `ch5` 负责回答“Agent、Pattern、Runner、Service 的边界是什么”
- `ch8` 负责回答“ReAct 主循环是怎么跑的、要改哪里”

因此最合理的做法是：

**以 `ch5` 约束边界，以 `ch8` 作为实际改造蓝本。**

### 2.3 对 VerticalAgent 的定位

`VerticalAgent` 不是本次架构改造的核心起点。

它更适合承担：

- 把造数领域的默认工具、默认 prompt、默认 pattern 打包为一个 agent 类型
- 让系统通过 `agent_type` 装配新的领域 agent

但它不能天然解决：

- 唯一控制律
- `NextActionDecision`
- Guard / Runtime / Ledger 分层
- Observation 回流

因此本次改造的核心应当是：

**新建一个造数领域 Pattern，而不是先围绕 VerticalAgent 设计。**

## 3. 本次改造的核心判断

### 3.1 可以复用 xagent 的地方

可以直接借用或延续的能力：

- `Agent` 作为轻量状态与配置容器
- `AgentContext` 作为运行态共享上下文
- `AgentRunner` 作为 pattern 调度器
- `AgentService` 作为任务生命周期与 Web 接入入口
- `ReActPattern` 的循环骨架
- `DAGPlanExecutePattern` 的执行稳定性能力
- `MemoryStore` 的 recall / 语义检索能力
- `Tracer` 的可观测性能力
- WebSocket 的聊天与交互 UI 通道
- SQL / HTTP API 等已有工具

### 3.2 不能直接照搬 xagent 的地方

以下部分必须重做或显著扩展：

- 现有 `react.Action`
  - 现在只有 `tool_call / final_answer`
  - 不满足 `decision_mode / action_kind / risk_level / requires_approval`
- 现有 `_execute_action()`
  - 当前是 Agent 直接碰工具
  - 不符合 `Agent -> Guard -> Runtime -> Resource`
- 现有 `MemoryStore`
  - 更适合 recall，不适合作为结构化 ledger
- 现有 DAG 主循环
  - 更适合作为 Runtime 内部执行器
  - 不适合作为唯一业务决策源
- 现有系统没有独立的审批通道和审批记录模型
- 现有系统没有资源目录与风险元数据层

## 4. xagent 当前基线能力与参考入口

以下代码是本次改造最重要的参考点。

### 4.1 Agent 基础边界

- `src/xagent/core/agent/agent.py`
  - `Agent` 只是配置与状态容器
- `src/xagent/core/agent/context.py`
  - `AgentContext` 维护运行时 `state/history/task_id`
- `src/xagent/core/agent/runner.py`
  - `AgentRunner` 负责调度 pattern，不承担业务决策
- `src/xagent/core/agent/service.py`
  - `AgentService` 负责工具初始化、任务执行、pause/resume、reconstruct

### 4.2 ReAct 主循环

- `src/xagent/core/agent/pattern/react.py`
  - 当前是最重要的改造蓝本
  - 可复用：
    - 结构化 JSON 输出
    - `_execute_react_loop()`
    - `_get_action_from_llm()`
    - `_execute_action()`
    - pause / resume / interrupt
    - trace / memory / compact 逻辑

### 4.3 DAG 执行器

- `src/xagent/core/agent/pattern/dag_plan_execute/dag_plan_execute.py`
- `src/xagent/core/agent/pattern/dag_plan_execute/plan_executor.py`
- `src/xagent/core/agent/pattern/dag_plan_execute/models.py`

这些代码更适合在新架构中作为：

- `Runtime / Workflow Plane` 的内部执行器
- 而不是外层唯一主脑

### 4.4 工具与资源入口

- `src/xagent/core/tools/adapters/vibe/factory.py`
- `src/xagent/core/tools/adapters/vibe/sql_tool.py`
- `src/xagent/core/tools/adapters/vibe/api_tool.py`

这些代码适合作为未来 `Resource Plane` 的底层能力来源。

### 4.5 记忆与可观测性

- `src/xagent/core/memory/base.py`
- `src/xagent/core/memory/core.py`
- `src/xagent/core/agent/trace.py`

其中：

- `MemoryStore` 适合做 recall / semantic memory
- `Tracer` 适合做执行证据流和调试事件
- 但结构化 ledger 仍需新增

### 4.6 Web 通道

- `src/xagent/web/api/websocket.py`

当前已具备：

- `chat_response`
- `interactions`
- continuation
- pause / resume 交互

因此用户澄清通道可以大量复用。

## 5. 你的流程图与 xagent 类级对照

下表按你的流程图与五层架构逐项对照到 xagent 代码类。

| 你的流程节点 / 设计概念 | 设计定位 | xagent 当前对应类 | 当前类主要负责什么 | 是否可直接复用 | 建议改造方向 |
|---|---|---|---|---:|---|
| `INIT（初始化上下文）` | 统一装配当前轮上下文 | `AgentContext` | 保存 `task_id/user_id/session_id/state/history` | 部分可复用 | 在 `state` 中新增 `flow_draft / ledger_snapshot / recall_snapshot / approvals` |
| `AGENT CONTROL PLANE` | 唯一业务决策源 | `ReActPattern` | 通过 LLM 做 `tool_call / final_answer` 循环 | 骨架可复用 | 新建 `DataMakeReActPattern` 替代现有动作语义 |
| `NextActionDecision` | 每轮统一决策契约 | `react.Action` | 结构化输出 `tool_call/final_answer` | 不可直接复用 | 新增 `NextActionDecision` 模型 |
| `ACTION DISPATCH` | 分发关节，无业务决策权 | `ReActPattern._execute_action()` | 直接调 tool 或结束 | 不能直接复用 | 改成 `interaction / supervision / execution / terminate` 分发 |
| `interaction_action` | 用户澄清、解释、候选确认 | `ChatResponse / Interaction`、`websocket.py` | 发送聊天消息和交互组件 | 大量可复用 | 抽象为 `InteractionBridge`，返回 `InteractionObservation` |
| `supervision_action` | 人工审批 / 修正 / 兜底 | 当前无独立类 | 无真正审批链路 | 基本不可复用 | 新增 `SupervisionBridge + ApprovalRecord` |
| `execution_action` | 系统执行入口 | 当前 `tool_call` | Agent 直接调用资源 | 不符合要求 | 变为 `Pattern -> Guard -> Runtime -> Resource` |
| `Guard / Routing Plane` | 合法性、风险、路由、审批要求 | 当前无独立层 | 仅有 tool schema / allowed_tools 等轻约束 | 不足 | 新增 `GuardService + GuardVerdict` |
| `Runtime / Workflow Plane` | 单动作稳定执行 | `DAGPlanExecutePattern / PlanExecutor` | 计划、step 执行、重试、暂停恢复 | 部分可复用 | 下沉为 Runtime 内部执行器 |
| `Resource Plane` | 真实资源调用层 | `sql_tool.py / api_tool.py / ToolFactory` | SQL、HTTP、文件等工具调用 | 大量可复用 | 在其上新增 `ResourceCatalog / ResourceAdapter` |
| `Memory / Ledger Plane` | 证据、恢复、审计 | `MemoryStore / Tracer` | recall、trace、事件恢复 | 仅部分可复用 | 新增 `LedgerRepository` 承载结构化账本 |
| `Observation 回流` | 所有非终止结果统一回流 | 当前 observation 多为工具结果字典 | 无统一 envelope | 不足 | 新增 `ObservationEnvelope` 与分类 payload |
| `REACT LOOP NEXT ROUND` | 读证据后二次决策 | `_execute_react_loop()` | 循环调 LLM -> action -> observation | 骨架可复用 | 改为从 ledger 读取统一证据 |

## 6. 建议新增的核心类与代码落点

以下是建议的第一版代码拆分。命名可后续微调，但职责边界建议尽量保持稳定。

### 6.1 Agent Control Plane

#### `DataMakeReActPattern`

- 建议位置：
  - `src/xagent/core/agent/pattern/data_make_react.py`
- 对应你的设计：
  - `Agent Control Plane`
- 主要职责：
  - 读取当前轮上下文
  - 聚合 recall + ledger snapshot + flow draft
  - 调 LLM 生成 `NextActionDecision`
  - 调用 `ActionDispatcher`
  - 接收 `ObservationEnvelope`
  - 结果回写 ledger
  - 进入下一轮决策

#### `NextActionDecision`

- 建议位置：
  - `src/xagent/core/agent/pattern/data_make_models.py`
- 对应你的设计：
  - `Agent Control Plane` 输出契约
- 主要职责：
  - 统一描述这一轮的动作决策
  - 关键字段建议包含：
    - `decision_mode`
    - `action_kind`
    - `action`
    - `reasoning`
    - `goal_delta`
    - `params`
    - `expected`
    - `risk_level`
    - `requires_approval`
    - `user_visible`
    - `final_status`
    - `final_message`

#### `ActionDispatcher`

- 建议位置：
  - 可先作为 `DataMakeReActPattern` 内部方法
  - 稳定后可抽为 `src/xagent/core/datamake/dispatcher.py`
- 对应你的设计：
  - `ACTION DISPATCH`
- 主要职责：
  - `decision_mode == terminate` 时终止
  - `interaction_action` 调 `InteractionBridge`
  - `supervision_action` 调 `SupervisionBridge`
  - `execution_action` 调 `GuardService`

### 6.2 Guard / Routing Plane

#### `GuardService`

- 建议位置：
  - `src/xagent/core/datamake/guard.py`
- 对应你的设计：
  - `Guard / Routing Plane`
- 主要职责：
  - 动作合法性检查
  - 参数完整性校验
  - 风险等级评估
  - 是否需要审批判定
  - 资源就绪性检查
  - 决定执行路由

#### `GuardVerdict`

- 建议位置：
  - `src/xagent/core/datamake/models.py`
- 对应你的设计：
  - `Guard / Routing Plane` 输出
- 主要职责：
  - 输出结构化裁决结果，例如：
    - `allowed`
    - `normalized_action`
    - `route`
    - `blockers`
    - `approval_required`
    - `readiness_snapshot`

### 6.3 Runtime / Workflow Plane

#### `RuntimeExecutor`

- 建议位置：
  - `src/xagent/core/datamake/runtime.py`
- 对应你的设计：
  - `Runtime / Workflow Plane`
- 主要职责：
  - 将批准后的 execution action 编译并执行
  - 统一处理：
    - retry
    - timeout
    - idempotency
    - pause / resume
  - 产出：

#### `ExecutionCompiler`

## 7. 推荐的改造起点与代码顺序

### 7.1 设计顺序

推荐按下面顺序推进：

1. 先基于 `ch5-base-agent-model` 锁定边界
2. 再基于 `ch8-react-pattern` 设计造数 ReAct 主循环
3. 然后新增 `Guard / Runtime / Ledger`
4. 最后再决定是否包成 `VerticalAgent`

### 7.2 代码顺序

推荐按下面顺序落代码：

1. 新增领域模型
   - `NextActionDecision`
   - `ObservationEnvelope`
   - `GuardVerdict`
   - `FlowDraft`
   - `DecisionFrame`

2. 新增主脑 pattern
   - `DataMakeReActPattern`

3. 新增执行分层
   - `ActionDispatcher`
   - `GuardService`
   - `RuntimeExecutor`

4. 新增资源层
   - `ResourceCatalog`
   - SQL / HTTP 资源适配器
   - Dubbo 资源适配器

5. 新增 ledger 层
   - `LedgerRepository`
   - 各类 run / approval / draft 模型

6. 新增 interaction / supervision bridge

7. 最后按需要接回：
   - `AgentService`
   - WebSocket
   - 持久化存储

## 8. 分阶段实施建议

### 第一阶段：最小可运行闭环（MVP）

目标：

- 跑通 `SQL + HTTP` 的智能造数闭环
- 建立新 ReAct 主循环
- 建立最小 ledger

范围建议：

- 新增 `DataMakeReActPattern`
- 新增 `NextActionDecision / ObservationEnvelope / GuardVerdict`
- 新增 `FlowDraft / ExecutionRun / DecisionFrame`
- 复用现有 SQL / HTTP tool
- 复用现有 WebSocket 交互
- 暂不接 Dubbo
- 暂不做完整审批系统，只保留 `confirm` 型 supervision

### 第二阶段：补齐控制层

目标：

- 真正建立 `Guard / Runtime / Approval` 体系

范围建议：

- 补齐 `GuardService`
- 支持 `probe` 与 `execute` 分离
- 补齐 `ApprovalRecord`
- 增加高风险动作审批
- 增加 `PauseObservation / ResumeToken`

### 第三阶段：沉淀场景资产与复用

目标：

- 让系统越来越依赖历史场景与模板，而不是每次纯从零推理

范围建议：

- 相似造数场景召回
- `FlowDraft -> 场景模板` 沉淀
- 稳定场景编译为静态 Graph / Template
- 高频链路采用：
  - 历史模板优先
  - Agent 决策校验
  - Runtime 执行

## 9. 风险与关键注意点

### 9.1 不能让 Pattern 直接碰资源

如果 `DataMakeReActPattern` 继续像现有 `ReActPattern` 一样直接调 tool，很容易退化回：

```text
Agent 直接控制工具调用
```

这与目标架构冲突。

### 9.2 不能把 DAG 继续当主脑

`DAGPlanExecutePattern` 的长处在于：

- 计划
- 并发
- 稳定执行
- pause / resume / continuation

它适合做 Runtime 内部执行器，但不适合作为唯一业务决策源。

### 9.3 不能把 MemoryStore 当 Ledger

`MemoryStore` 适合 recall，不适合做强结构化业务账本。

Ledger 至少需要能稳定表达：

- 当前工作草稿
- 决策原因
- 审批记录
- probe / execution 结果
- 恢复依据

### 9.4 Dubbo 是明确新增项

当前代码中未发现 Dubbo 资源接入能力，因此这部分不能按“现成复用”估算。

### 9.5 高风险动作必须有 Guard 与审批

造数平台天然涉及：

- SQL 写操作
- 跨系统 HTTP 调用
- Dubbo 服务调用
