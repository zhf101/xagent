# 智能造数平台基于 xagent 的代码类结构设计 v3

## 1. 文档目的

本文不展开核心数据模型字段，只解决一件事：

**把“智能造数平台架构设计 v7”对应成一版清晰、可落地、可核查进度的代码类结构。**

这样后续可以直接按：

- 目录是否已创建
- 类是否已落地
- 类职责是否符合分层
- 层间调用是否越界

来检查实现进度，而不是只看概念。

---

## 2. 本版范围

本版只覆盖：

- 代码分层
- 类结构
- 目录结构
- 层间依赖关系
- 与 xagent 现有类的映射
- 进度核查清单

本版暂不覆盖：

- 核心数据模型字段级定义
- 表结构与持久化 schema
- 详细伪代码
- 测试样例代码

## 3. 设计原则

### 3.1 总原则

代码结构必须直接映射你的架构分层：

```text
Agent Control Plane
Guard / Routing Plane
Runtime / Workflow Plane
Resource Plane
Memory / Ledger Plane
+ User / UI Channel
+ Human in Loop Channel
```

### 3.2 结构目标

本次代码结构要满足 4 个目标：

1. **按层可读**
   - 一眼能看出每个类属于哪一层

2. **按职责可拆**
   - 类职责清晰，后续实现时不容易边写边串层

3. **按进度可核查**
   - 可以按“目录 -> 文件 -> 类”逐项核查

4. **按边界可约束**
   - 避免出现：
     - Pattern 直接调底层资源
     - Ledger 反过来驱动主流程
     - Runtime 自己决定业务下一步

## 4. 总体代码分层图

建议把新增造数领域代码统一收敛到一个独立目录，避免分散在现有通用 agent 目录中难以核查。

### 4.1 目标目录结构

```text
src/xagent/
└── core/
    ├── agent/
    │   └── pattern/
    │       └── data_make_react.py              # 顶层主脑 Pattern
    │
    └── datamake/
        ├── application/
        │   ├── dispatcher.py                   # Action Dispatch
        │   ├── interaction.py                  # User / UI bridge
        │   ├── supervision.py                  # Human-in-loop bridge
        │   └── orchestrator.py                 # 可选：应用服务编排层
        │
        ├── guard/
        │   ├── service.py                      # GuardService
        │   ├── policy.py                       # 风险策略 / 审批策略
        │   └── readiness.py                    # 资源就绪性检查
        │
        ├── runtime/
        │   ├── executor.py                     # RuntimeExecutor
        │   ├── compiler.py                     # ExecutionCompiler
        │   ├── probe.py                        # Probe 执行逻辑
        │   ├── execution.py                    # Execute 执行逻辑
        │   └── recovery.py                     # pause / resume / token 恢复
        │
        ├── resources/
        │   ├── catalog.py                      # ResourceCatalog
        │   ├── registry.py                     # 资源注册与查找
        │   ├── sql_adapter.py                  # SQL 资源适配器
        │   ├── http_adapter.py                 # HTTP 资源适配器
        │   ├── dubbo_adapter.py                # Dubbo 资源适配器
        │   └── verifier.py                     # 资源级探测与验证
        │
        ├── ledger/
        │   ├── repository.py                   # LedgerRepository
        │   ├── snapshots.py                    # 恢复快照构建
        │   ├── projections.py                  # 状态投影
        │   └── replay.py                       # 回放 / 恢复视图
        │
        ├── contracts/
        │   ├── decision.py                     # NextActionDecision 等契约
        │   ├── observation.py                  # ObservationEnvelope 等契约
        │   ├── guard.py                        # GuardVerdict 等契约
        │   ├── interaction.py                  # interaction / approval ticket 契约
        │   └── runtime.py                      # compiled contract / runtime result 契约
        │
        └── services/
            ├── recall_service.py               # Recall / memory 查询服务
            ├── draft_service.py                # FlowDraft 读写服务
            └── approval_service.py             # 审批业务服务
```

### 4.2 为什么建议单独建 `datamake/`

原因有三个：

1. 你的设计已经超出普通 pattern 改造，属于一个完整领域子系统
2. 后续核查实现进度时，不希望代码分散在 `agent/`、`web/`、`tools/` 多处难以汇总
3. 能明确区分：
   - xagent 通用底座
   - datamake 领域控制内核

## 6. Agent Control Plane

### 6.1 层职责

对应你的设计：

- 唯一业务决策源
- 每一轮根据：
  - 当前输入
  - recall
  - flow draft
  - ledger snapshot
  - execution / interaction / supervision observation
 重新决定下一步最值得做什么

### 6.2 主要类

#### `DataMakeReActPattern`

- 文件：
  - `src/xagent/core/agent/pattern/data_make_react.py`
- 当前层：
  - `Agent Control Plane`
- 在你的设计里属于：
  - `AGENT CONTROL PLANE`
- 主要职责：
  - 读取运行时上下文
  - 读取 Ledger 快照
  - 调用 Recall 查询
  - 构造本轮 Agent 输入
  - 调用 LLM 生成 `NextActionDecision`
  - 调用 `ActionDispatcher`
  - 接收 `ObservationEnvelope`
  - 将结果写回 Ledger
  - 决定进入下一轮还是 terminate

### 6.3 辅助类

#### `DecisionBuilder`

- 文件建议：
  - `src/xagent/core/datamake/application/orchestrator.py`
- 当前层：
  - `Agent Control Plane` 辅助组件
- 主要职责：
  - 负责把：
    - FlowDraft
    - RecallSnapshot
    - Recent Observations
    - Pending Ticket
  组装成一轮可供 LLM 推理的上下文视图

#### `TerminationResolver`

- 文件建议：
  - `src/xagent/core/datamake/application/orchestrator.py`
- 当前层：
  - `Agent Control Plane` 辅助组件
- 主要职责：
  - 统一判断 terminate 类型结果如何落账本、如何返回最终输出

### 6.4 本层与 xagent 现有类关系

| xagent 现有类 | 关系 |
|---|---|
| `Agent` | 继续作为容器，不承载主流程决策 |
| `AgentPattern` | `DataMakeReActPattern` 的抽象基类 |
| `ReActPattern` | 参考其主循环骨架，但不复用其 `tool_call/final_answer` 动作语义 |
| `AgentRunner` | 继续负责调 pattern，不承担业务控制 |

### 6.5 本层不应该做什么

- 不直接执行 SQL / HTTP / Dubbo
- 不直接管理审批状态机
- 不直接决定资源调用细节
- 不直接承担恢复快照拼装逻辑

## 7. Action Dispatch

### 7.1 层职责

对应你的设计：

- 只是投递关节
- 没有业务决策权
- 先判断 `decision_mode`
- 再判断 `action_kind`

### 7.2 主要类

#### `ActionDispatcher`

- 文件：
  - `src/xagent/core/datamake/application/dispatcher.py`
- 当前层：
  - 应用调度层
- 在你的设计里属于：
  - `ACTION DISPATCH`
- 主要职责：
  - 如果 `decision_mode == terminate`
    - 直接返回终止结果
  - 如果 `action_kind == interaction_action`
    - 调 `InteractionBridge`
  - 如果 `action_kind == supervision_action`
    - 调 `SupervisionBridge`
  - 如果 `action_kind == execution_action`
    - 调 `GuardService`

### 7.3 本层不应该做什么

- 不判断“是否应该 terminate”
- 不修改 Agent 目标
- 不替 Guard 做校验
- 不替 Runtime 做执行

## 9. Runtime / Workflow Plane

### 9.1 层职责

对应你的设计：

- 单动作稳定执行
- retry / timeout / idempotency
- pause / resume
- probe / execute 分离
- 输出结构化执行结果

### 9.2 第一阶段结构

第一阶段建议只做“单资源、单动作、受控执行”。

因此第一阶段 Runtime 不必一开始就引入 DAG。

### 9.3 主要类

#### `RuntimeExecutor`

- 文件：
  - `src/xagent/core/datamake/runtime/executor.py`
- 当前层：
  - `Runtime / Workflow Plane`
- 在你的设计里属于：
  - `Runtime / Workflow Plane`
- 主要职责：
  - 接收 `GuardVerdict`
  - 调用 `ExecutionCompiler`
  - 统一执行 probe / execute
  - 处理 timeout / retry / idempotency
  - 输出 execution/failure/pause observation

#### `ExecutionCompiler`

- 文件：
  - `src/xagent/core/datamake/runtime/compiler.py`
- 当前层：
  - `Runtime / Workflow Plane`
- 主要职责：
  - 把 execution action 编译成内部执行契约
  - 保证 Runtime 面对的是标准化动作，而不是原始自由文本

#### `ProbeExecutor`

- 文件：
  - `src/xagent/core/datamake/runtime/probe.py`
- 当前层：
  - `Runtime / Workflow Plane`
- 主要职责：
  - 执行 `probe` 模式动作
  - 验证参数、资源连通性、返回结构等

#### `ActionExecutor`

- 文件：
  - `src/xagent/core/datamake/runtime/execution.py`
- 当前层：
  - `Runtime / Workflow Plane`
- 主要职责：
  - 执行正式 `execute` 动作

#### `RecoveryCoordinator`

- 文件：
  - `src/xagent/core/datamake/runtime/recovery.py`
- 当前层：
  - `Runtime / Workflow Plane`
- 主要职责：
  - 协调 pause / resume token
  - 为长动作恢复提供技术支持

### 9.4 第二阶段可接入类

#### `DagRuntimeAdapter`

- 文件建议：
  - `src/xagent/core/datamake/runtime/dag_adapter.py`
- 当前层：
  - `Runtime / Workflow Plane`
- 主要职责：
  - 将复杂 execution_action 下沉到现有 `DAGPlanExecutePattern`

### 9.5 本层与 xagent 现有类关系

| xagent 现有类 | 关系 |
|---|---|
| `DAGPlanExecutePattern` | 第二阶段作为复杂 Runtime 内部执行器 |
| `PlanExecutor` | 复杂执行时可复用其并发与依赖调度能力 |
| `AgentService.pause_execution` | 可继续借用整体任务级暂停入口 |

### 9.6 本层不应该做什么

- 不决定业务下一步
- 不直接和用户对话
- 不自己创建审批流

## 14. Contracts 层

虽然本版不写字段级模型，但为了代码分层清晰，仍建议单独设一个 `contracts/` 目录。

### 14.1 目录职责

```text
contracts/
  decision.py      # 决策类契约
  observation.py   # observation 类契约
  guard.py         # guard verdict 类契约
  interaction.py   # interaction / approval ticket 契约
  runtime.py       # runtime compiled contract / run result 契约
```

### 14.2 为什么要单独拆 contracts

原因：

1. 让各层都依赖统一契约，而不是相互 import 具体实现类
2. 后续核查实现进度时，可以单独检查契约是否稳定
3. 避免模型类散落在不同层里难以维护

```text
                       SQL Adapter HTTP Adapter Dubbo Adapter
```

### 15.1 允许依赖

- `DataMakeReActPattern -> ActionDispatcher`
- `ActionDispatcher -> InteractionBridge / SupervisionBridge / GuardService`
- `GuardService -> ResourceCatalog`
- `RuntimeExecutor -> ResourceCatalog`
- `InteractionBridge / SupervisionBridge / RuntimeExecutor -> LedgerRepository`
- `DataMakeReActPattern -> LedgerRepository / RecallService`

### 15.2 不允许依赖

- `LedgerRepository -> ActionDispatcher`
- `ResourceCatalog -> DataMakeReActPattern`
- `RuntimeExecutor -> InteractionBridge`
- `SupervisionBridge -> RuntimeExecutor`

## 19. 最终建议

这版代码类结构设计的核心目标，是把你的架构图真正落成：

**按层可见、按类可查、按边界可约束。**

如果后续要核查实现进度，建议只按下面三步看：

1. 先看目录有没有齐
2. 再看每层关键类有没有齐
3. 再看层间依赖有没有越界

一句话概括：

**先把类结构站稳，再补数据模型和伪代码，这样后面实现时最不容易失控。**
