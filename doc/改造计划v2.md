# 智能造数平台基于 xagent 的改造计划 v2

## 1. 文档目的

本文是在 [改造计划v1.md](D:/git/codebase/dev0330/xagent/doc/改造计划v1.md) 基础上的补强版。

`v1` 已经完成：

- 改造起点选择
- 你的五层架构与 xagent 的类级对照
- 建议新增的核心类与落点
- 分阶段实施建议

但在工程评审中，`v1` 暴露出 4 个会直接影响落地的结构性缺口：

1. `interaction_action / supervision_action` 的挂起、恢复、幂等协议未定义
2. `LedgerRepository` 的主存证模型与唯一事实源未定义
3. `ResourceCatalog + GuardService` 的安全边界未定义
4. 第一阶段 MVP 的裁剪原则与测试矩阵未定义

本文只解决这 4 个问题，为后续进入数据模型草案与主循环伪代码做准备。

---

## 2. 本版结论

### 2.1 保持不变的主线

整体改造主线保持不变：

```text
先用 ch5 锁定边界
再以 ch8 的 ReAct 作为主改造蓝本
新增造数领域控制 Pattern
补 Guard / Runtime / Ledger / ResourceCatalog
最后按需要再包成 VerticalAgent
```

### 2.2 本版新增的关键工程决定

本版新增四个关键决策：

1. **用户澄清与人工审批默认采用“持久化等待态 + 后续 continuation 恢复”**
   - 不建议让顶层 ReAct 协程长期阻塞等待人回复
   - 推荐把当前轮决策与上下文落账本，任务进入 `waiting_user / waiting_human`
   - 后续由新的用户输入或人工审批结果触发下一轮恢复

2. **Ledger 是业务唯一事实源，Tracer 只是诊断证据流，MemoryStore 只是 recall**
   - 恢复、审计、重放、状态投影一律读 Ledger
   - Tracer 不参与业务恢复判定
   - MemoryStore 不承载业务状态

3. **Execution Action 不能直接变成任意 SQL / HTTP / Dubbo 文本执行**
   - 必须先落到 `ResourceCatalog` 注册的资源与动作模板
   - LLM 只能在“受控动作空间”里选择与补参

4. **第一阶段只做“单资源执行 + 明确受控动作”，不引入 DAG Runtime**
   - `DAGPlanExecutePattern` 先下沉为第二阶段能力
   - 第一阶段先跑通最小闭环，避免平台化过度设计

## 3. 关键缺口一：Interaction / Supervision 的挂起与恢复协议

这是当前最先要补齐的部分，因为它直接决定：

- `DataMakeReActPattern` 遇到澄清问题时怎么停
- 用户回复后如何恢复到正确轮次
- 人工审批如何关联到原始决策
- 系统重启或 WebSocket 断线后如何续跑

### 3.1 推荐策略

推荐采用下面的恢复机制：

```text
Agent 决策出 interaction_action / supervision_action
-> 创建 Ticket（待用户 / 待人工）
-> 将 Ticket + Decision + FlowDraft Snapshot 持久化到 Ledger
-> 产生 PauseObservation
-> 当前执行轮结束，任务状态进入 waiting_user 或 waiting_human

用户/人工稍后回复
-> 外部通道写入 InteractionObservation / SupervisionObservation
-> 再次触发 DataMakeReActPattern
-> 从 Ledger 恢复最近未完成 Ticket 对应的上下文
-> 进入下一轮决策
```

### 3.2 为什么不建议顶层协程长时间阻塞等待

不建议采用：

```text
在 DataMakeReActPattern 内部 await 用户输入
```

原因：

- WebSocket 断连后恢复复杂
- 长协程生命周期不利于稳定性
- 容易让业务状态留在内存里而不是账本里
- 与你强调的“Ledger 是恢复平面”冲突

因此推荐：

**等待态落库，下一条消息继续。**

### 3.3 需要新增的协议对象

建议新增两个 ticket 模型：

#### `InteractionTicket`

主要字段建议：

- `ticket_id`
- `task_id`
- `session_id`
- `round_id`
- `decision_id`
- `decision_action`
- `status`
  - `pending`
  - `answered`
  - `expired`
  - `cancelled`
- `questions`
- `expected_signals`
- `flow_draft_snapshot`
- `created_at`
- `expired_at`

#### `ApprovalTicket`

主要字段建议：

- `approval_id`
- `task_id`
- `session_id`
- `round_id`
- `decision_id`
- `risk_level`
- `approval_scope`
- `requested_by`
- `status`
  - `pending`
  - `approved`
  - `rejected`
  - `expired`
  - `cancelled`
- `action_payload`
- `flow_draft_snapshot`
- `created_at`
- `expired_at`

### 3.4 需要统一的关联 ID

后续所有交互与审批都建议统一携带以下关联键：

- `task_id`
- `session_id`
- `round_id`
- `decision_id`
- `ticket_id / approval_id`

含义建议：

- `task_id`
  - 任务级唯一标识
- `session_id`
  - 会话级唯一标识
- `round_id`
  - Agent 每一轮决策循环编号
- `decision_id`
  - 每条 `NextActionDecision` 的唯一标识
- `ticket_id / approval_id`
  - 当前等待中的用户澄清 / 人工审批实体

### 3.5 需要新增的状态投影

建议在任务状态层增加以下业务态：

- `running`
- `waiting_user`
- `waiting_human`
- `paused_system`
- `completed`
- `failed`
- `cancelled`

这里要注意：

- 这些状态是**投影**
- 真正恢复仍然依赖 Ledger 中的事件和快照
- 不能让状态投影反过来驱动业务主循环

### 3.6 对应到 xagent 的类级改造

| 你的设计 | 建议类 | 功能 |
|---|---|---|
| `interaction_action` | `InteractionBridge` | 发送用户问题、创建 `InteractionTicket`、回收用户回答 |
| `supervision_action` | `SupervisionBridge` | 发送审批请求、创建 `ApprovalTicket`、回收审批结果 |
| `PauseObservation` | `ObservationEnvelope` | 表示当前轮进入等待态 |
| `下一轮恢复` | `DataMakeReActPattern` | 读取 Ledger 中待完成 ticket 对应上下文并继续运行 |
| `Web 接入` | `AgentService + websocket.py` | 接收入站消息并转为 continuation 请求 |

### 3.7 推荐的恢复流程图

```text
┌─────────────────────────────────────────┐
│ round N: NextActionDecision            │
│ action_kind = interaction_action       │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│ InteractionBridge                       │
│ 1. create InteractionTicket             │
│ 2. persist ticket + draft snapshot      │
│ 3. emit PauseObservation                │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│ Task Status Projection                  │
│ waiting_user                            │
└─────────────────┬───────────────────────┘
                  │ 用户稍后回复
                  ▼
┌─────────────────────────────────────────┐
│ websocket / api input                   │
│ 1. write InteractionObservation         │
│ 2. mark ticket answered                 │
│ 3. trigger continuation                 │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│ DataMakeReActPattern                    │
│ 读取 Ledger 恢复上下文，进入 round N+1   │
└─────────────────────────────────────────┘
```

## 4. 关键缺口二：Ledger 的主存证模型与唯一事实源

这是当前架构能否稳定恢复和审计的核心。

### 4.1 统一规则

建议明确三条规则：

1. **Ledger 是业务唯一事实源**
2. **Tracer 是诊断和可观测性通道，不是业务恢复源**
3. **MemoryStore 只负责 recall，不负责业务状态**

### 4.2 三层数据职责分工

#### A. Ledger（结构化业务账本）

负责：

- 决策记录
- observation 记录
- 当前工作草稿
- 审批记录
- 执行记录
- 恢复依据

典型用途：

- 恢复
- 审计
- 回放
- 风险追踪
- 状态投影

#### B. Tracer（诊断事件流）

负责：

- 调试
- 可观测性
- 页面执行过程展示
- LLM / tool / runtime 事件采集

典型用途：

- 调试控制台
- 过程展示
- 性能分析

#### C. MemoryStore（语义记忆）

负责：

- 相似造数场景召回
- 历史经验 recall
- prompt 增强

典型用途：

- 召回相似模板
- 回忆历史执行经验

### 4.3 Ledger 需要的基础对象

建议 Ledger 最少包含以下实体。

#### `FlowDraft`

职责：

- 当前任务的工作理解主对象
- 是“当前理解”的主投影，不是 append-only 日志

建议字段：

- `draft_id`
- `task_id`
- `goal`
- `confirmed_inputs`
- `open_questions`
- `candidate_steps`
- `chosen_strategy`
- `risk_summary`
- `status_projection`
- `version`
- `updated_at`

#### `DecisionFrame`

职责：

- 记录每一轮 Agent 为什么做出当前决策
- 建议 append-only

建议字段：

- `decision_id`
- `task_id`
- `round_id`
- `draft_version`
- `decision_mode`
- `action_kind`
- `action`
- `reasoning_summary`
- `goal_delta`
- `risk_level`
- `requires_approval`
- `created_at`

#### `ObservationRecord`

职责：

- 持久化所有 ObservationEnvelope
- 建议 append-only

建议字段：

- `observation_id`
- `task_id`
- `round_id`
- `decision_id`
- `observation_type`
- `status`
- `actor_type`
- `summary`
- `payload`
- `evidence`
- `created_at`

#### `ExecutionRun`

职责：

- 记录正式执行或试探执行的技术结果
- 可按 run 维度聚合多个 step

建议字段：

- `run_id`
- `task_id`
- `decision_id`
- `mode`
  - `probe`
  - `execute`
- `resource_targets`
- `compiled_contract`
- `status`
- `result_summary`
- `error_summary`
- `resume_token`
- `started_at`
- `finished_at`

#### `ApprovalRecord`

职责：

- 审批与人工修正记录
- append-only

#### `ResumeToken`

职责：

- 记录恢复时需要的关键引用，而不是大块冗余上下文

### 4.4 建议的持久化形式

建议采用：

- **append-only 事件表**
  - `decision_frames`
  - `observation_records`
  - `approval_records`
  - `execution_runs`

- **可更新投影表**
  - `flow_drafts`
  - `task_state_projection`

这比把所有状态都堆在一张大 JSON 表里更稳。

### 4.5 恢复时的优先级

恢复顺序建议如下：

```text
1. 读取 FlowDraft 当前投影
2. 读取最近未完成的 DecisionFrame / Ticket / Approval
3. 读取相关 ObservationRecord / ExecutionRun
4. 拼出当前轮上下文视图
5. 再让 DataMakeReActPattern 进入下一轮
```

不建议：

```text
仅依赖 Tracer 反推业务状态
```

### 4.6 LedgerRepository 的职责边界

`LedgerRepository` 应只负责：

- 写入记录
- 查询记录
- 构造恢复视图
- 提供只读上下文快照

`LedgerRepository` 不应负责：

- 决定下一步动作
- 自动推进业务状态机
- 替 Agent 做规划

## 5. 关键缺口三：ResourceCatalog 与 Guard 的安全边界

这是“智能造数平台”最危险、也最不能模糊的地方。

### 5.1 核心原则

必须避免下面这种退化：

```text
LLM 直接生成任意 SQL / 任意 URL / 任意 Dubbo 方法名
然后系统原样执行
```

推荐的原则是：

**LLM 只能在受控动作空间中做选择和补参，不能直接拥有原始资源执行权。**

### 5.2 ResourceCatalog 的建议形态

不建议把 `ResourceCatalog` 只做成“工具清单”。

建议做成：

```text
资源定义 + 动作模板 + 风险元数据 + 执行边界
```

建议每个资源动作最少包含：

- `resource_key`
- `resource_type`
  - `sql`
  - `http`
  - `dubbo`
- `operation_key`
- `display_name`
- `description`
- `env_scope`
- `write_mode`
  - `read_only`
  - `probe_only`
  - `write`
- `risk_level`
- `supports_probe`
- `requires_approval`
- `timeout_seconds`
- `retry_policy`
- `input_schema`
- `output_schema`
- `credential_ref`

### 5.3 三类资源的安全收敛方式

#### SQL

第一阶段建议不要开放任意 SQL 文本执行。

建议第一阶段仅支持：

- 已注册的 SQL 模板
- 参数化输入
- 明确声明读写属性

例如：

- `create_order_seed`
- `reset_order_status`
- `query_user_by_mobile`

而不是：

```text
让模型自由生成 update/delete/insert SQL 然后执行
```

#### HTTP

第一阶段建议不要开放任意 URL。

建议仅支持：

- Catalog 中注册的 endpoint
- 允许的方法
- 参数 schema
- 固定 credential 绑定

例如：

- `order_service.create_order`
- `user_service.bind_coupon`

#### Dubbo

第一阶段建议比 HTTP 还要保守。

建议仅支持：

- 已注册服务接口
- 已注册方法签名
- 已注册请求结构
- 明确环境与权限边界

### 5.4 GuardService 的职责细化

`GuardService` 至少要做 5 类判断：

1. **动作合法性**
   - 当前 action 是否存在
   - action 是否允许在当前环境执行

2. **参数完整性**
   - 必填参数是否齐全
   - 参数类型是否通过 schema 校验

3. **风险评估**
   - 是否写操作
   - 是否跨系统
   - 是否影响真实数据

4. **审批要求**
   - 是否必须人工确认
   - 是否可先 probe

5. **路由选择**
   - 单资源执行
   - 复合执行
   - 直接阻断

### 5.5 建议的安全执行链

```text
LLM 生成 NextActionDecision
-> ActionDispatcher 识别 execution_action
-> GuardService 根据 ResourceCatalog 做裁决
-> RuntimeExecutor 执行编译后的安全契约
-> ResourceAdapter 真正调用资源
```

### 5.6 凭证边界

建议明确一条规则：

**模型不能直接接触明文凭证。**

应采用：

- `credential_ref`
- 服务端注入凭证
- Runtime / Adapter 层读取凭证

而不是把 token/password 暴露给 Agent 上下文。

### 5.7 第一阶段推荐的动作空间

第一阶段建议把 execution_action 限定为以下类型：

- `probe_registered_action`
- `execute_registered_action`
- `verify_registered_action_result`

不建议第一阶段开放：

- 任意 SQL 生成执行
- 任意 URL 调用
- 任意 Dubbo 方法发现和执行
- 多资源自动编排

## 6. 关键缺口四：MVP 裁剪原则与测试矩阵

这是避免第一阶段失控的关键。

### 6.1 第一阶段的明确边界

第一阶段只做：

- 一个新的 `DataMakeReActPattern`
- 一个最小 `LedgerRepository`
- 一个最小 `GuardService`
- 一个最小 `RuntimeExecutor`
- 一个受控 `ResourceCatalog`
- SQL / HTTP 两类已注册动作
- 用户澄清型 `interaction_action`
- 简单确认型 `supervision_action`

第一阶段不做：

- Dubbo 执行
- DAG Runtime 复合执行
- 多资源自动编排
- 模板沉淀与 Graph 编译
- 复杂审批链
- 细粒度权限体系

### 6.2 第一阶段的执行原则

推荐把第一阶段的 execution action 限定为：

```text
一次 decision 最多驱动一个已注册资源动作
```

这样第一阶段主链路是：

```text
用户需求
-> Agent 澄清
-> Agent 选择一个已注册动作
-> Guard 校验
-> Runtime 执行
-> Observation 回流
-> Agent 再决定下一步
```

而不是一开始就做：

```text
一个决策自动拆成多个跨系统动作并编排执行
```

### 6.3 第一阶段测试矩阵

建议把测试分成 5 组。

#### A. `DataMakeReActPattern` 测试

覆盖点：

- 能正确解析 `NextActionDecision`
- `decision_mode = terminate` 时能结束
- `interaction_action` 能转为等待态
- `execution_action` 能正确流向 Guard
- 非法 JSON / 缺字段 / 错误 action_kind 时能给出失败 observation

#### B. `GuardService` 测试

覆盖点：

- 非注册动作被阻断
- 缺参数被阻断
- 高风险动作要求审批
- probe / execute 路由正确
- 写动作在只读环境下被阻断

#### C. `InteractionBridge / SupervisionBridge` 测试

覆盖点：

- 正确创建 ticket
- 重复回复只消费一次
- 超时 ticket 不可继续
- WebSocket 重连后可恢复
- interaction / approval 的 observation 能正确落账本

#### D. `RuntimeExecutor` 测试

覆盖点：

- 单资源动作能被正确执行
- timeout / retry 行为正确
- 同一 `decision_id` 重试时不重复执行写操作
- 失败时返回 `FailureObservation`
- 执行中断时返回 `PauseObservation`

#### E. `LedgerRepository` 测试

覆盖点：

- 能写入并恢复 `FlowDraft`
- 能基于 `DecisionFrame + ObservationRecord` 恢复当前轮上下文
- 多轮决策后能正确拿到最近未完成 ticket
- Tracer 缺失时仍可恢复

### 6.4 推荐的测试层次

第一阶段建议：

- **单元测试**
  - `DataMakeReActPattern`
  - `GuardService`
  - `RuntimeExecutor`
  - `LedgerRepository`

- **集成测试**
  - 从用户输入到 ticket 创建
  - 从 ticket 回答到 continuation 恢复
  - 从 decision 到 Guard 到 Runtime 到 Observation 回流

- **关键路径冒烟测试**
  - 一个 SQL 动作闭环
  - 一个 HTTP 动作闭环
  - 一个需要用户澄清后继续的闭环
  - 一个需要人工确认后继续的闭环

### 6.5 建议补的失败模式表

| 新代码路径 | 可能的真实失败 | 测试是否必须第一阶段覆盖 | 用户是否容易无感失败 |
|---|---|---:|---:|
| `DataMakeReActPattern` 解析 decision | LLM 返回错误 JSON | 是 | 是 |
| `InteractionBridge` 创建 ticket | ticket 已写入但消息未发到前端 | 是 | 是 |
| `SupervisionBridge` 回收审批 | 审批重复提交 | 是 | 否 |
| `GuardService` 校验资源 | catalog 配置与运行环境不一致 | 是 | 是 |
| `RuntimeExecutor` 执行写动作 | timeout 后重复执行导致重复写 | 是 | 否 |
| `LedgerRepository` 恢复 | 事件写了但投影未更新 | 是 | 是 |

对于满足下面三条的失败模式，必须视为第一阶段关键缺口：

1. 没有测试
2. 没有错误处理
3. 用户会静默失败

## 7. v2 后的类级职责更新

这一节只补充 `v1` 中缺少的职责边界。

### 7.1 `DataMakeReActPattern`

在 `v1` 基础上再补两条职责：

- 识别当前任务是否处于 `waiting_user / waiting_human`
- 从 Ledger 恢复最近未完成 ticket 对应的上下文快照

### 7.2 `InteractionBridge`

新增职责：

- 创建 `InteractionTicket`
- 写入 `PauseObservation`
- 将任务投影状态切为 `waiting_user`
- 防重复消费同一 ticket 回答

### 7.3 `SupervisionBridge`

新增职责：

- 创建 `ApprovalTicket`
- 将任务投影状态切为 `waiting_human`
- 回收审批结果并生成 `SupervisionObservation`

### 7.4 `LedgerRepository`

新增职责边界：

- 提供 `build_runtime_snapshot(task_id)` 能力
- 提供 `load_pending_ticket(task_id)` 能力
- 提供 `append_decision / append_observation / append_run / append_approval` 能力
- 不承担业务推进职责

### 7.5 `GuardService`

新增职责边界：

- 不接受“原始自由文本 SQL/HTTP/Dubbo”作为执行对象
- 只接受 `ResourceCatalog` 中注册的动作及其参数

### 7.6 `RuntimeExecutor`

第一阶段新增边界：

- 只执行“单资源、单动作、已注册”的执行契约
- 暂不支持 DAG 复合执行

## 9. 本版最终建议

`v1` 已经回答了“要改什么”，
`v2` 重点补上了“系统怎么稳定活下来”。

本版之后，整体路线可以进一步收敛成：

```text
第一阶段：
  只做单资源受控动作
  只做 SQL / HTTP
  只做最小等待态恢复
  只做最小 Ledger

第二阶段：
  再补审批增强、Dubbo、DAG Runtime、多资源编排
```

一句话概括：

**第一阶段先做“受控闭环”，不要先做“完整平台”。**
