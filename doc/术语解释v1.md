# 智能造数平台术语对照表

> 本文档对应《需求设计v1.md》中使用的所有英文术语，按功能层次分类整理。

---

## 一、核心概念层（Core Concepts）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **ReAct** | 推理-行动循环 | Reasoning and Acting 的缩写，一种AI推理模式，通过"推理→行动→观察→再推理"的循环完成任务。在系统中特指 Agent 的主循环机制。 |
| **Human in Loop** | 人工介入/人在回路 | 指人工参与和监督 AI 决策过程的机制。不是让 AI 完全自主，而是在关键节点引入人工确认、修正或兜底。 |
| **Agent** | 智能体 | 系统的"主脑"，唯一业务决策源。负责理解用户目标、选择下一步动作、决定是交互还是执行。 |
| **Guard** | 护栏/防护层 | 负责安全检查和准入控制，决定"这件事能不能做、该怎么安全地做"。可以阻断但不可以改写业务目标。 |
| **Runtime** | 运行时 | 负责"这件事怎样稳定地做完"，管理执行准备、重试、超时、幂等、暂停恢复等。 |
| **Resource** | 资源层 | 真正被调用并返回原始结果的底层能力，包括 SQL、HTTP、Dubbo、文件、队列、MCP 等。 |
| **Memory** | 共享记忆 | 长期存储和上下文管理，用于召回历史经验。 |
| **Ledger** | 账本 | 操作记录和审计日志，记录"为什么这样做、做到哪了、结果是什么"。 |

---

## 二、控制平面层（Control Planes）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **Agent Control Plane** | Agent 决策层/智能体控制平面 | 唯一业务决策源，负责理解用户目标、汇总上下文、建立任务假设、选择下一最优动作。只有这一层能决定"现在最值得做什么"。 |
| **Guard / Routing Plane** | 路由护栏层 | 负责动作归一化、参数检查、合法性校验、风险判断、权限检查、执行通道路由。决定"这件事能不能做、该怎么安全地做"。 |
| **Runtime / Workflow Plane** | 标准化执行层 | 负责单动作执行准备、原子动作执行、编译后契约执行、重试/超时/幂等控制、暂停/恢复。决定"这件事怎样稳定地做完"。 |
| **Resource Plane** | 资源层 | 提供工具、接口、文件、队列、MCP、SQL/HTTP/Dubbo/浏览器/数据源等真实能力。负责"真正被调用并返回原始结果"。 |
| **Memory / Ledger Plane** | 共享记忆/账本层 | 贯穿前四层的共享状态平面，不是串行层，而是证据/恢复/审计层。与各层都是"读/写证据"的关系。 |

---

## 三、交互通道层（Channels）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **User / UI Channel** | 用户/界面通道 | 承接解释、澄清、候选展示、用户确认的界面接口。只与 Agent Control Plane 交互，不应直接接管 Guard/Runtime。 |
| **Human in Loop Channel** | 人工介入监督通道 | 承接审批、人工确认、人工修正、人工兜底的监督通道。是外部通道，不是第六控制平面。 |

---

## 四、动作决策层（Action Decisions）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **NextActionDecision** | 下一动作决策 | Agent 每一轮必须输出的结构化决策协议，不是自由文本。包含 decision_mode、action_kind、action、params、risk 等字段。 |
| **Decision Mode** | 决策模式 | 决定当前轮是继续进入动作分发，还是直接终止。两种取值：`action`（进入动作分发）或 `terminate`（终止）。 |
| **Action Kind** | 动作类别 | 只在 `decision_mode=action` 时有效，决定下游走哪条通道。三种取值：`interaction_action`（交互）、`execution_action`（执行）、`supervision_action`（监督）。 |
| **Action Dispatch** | 动作分发 | 投递关节，无业务决策权。负责把 Agent 的决策送到正确通道，识别终止型决策并短路执行链。 |
| **Decision Frame** | 决策帧 | 单轮决策的记录单元，记录"为什么这样决策"。 |
| **Decision Ledger** | 决策账本 | 记录所有 Decision Frame 的持久化存储，用于证据存储、恢复支持、审计追溯。 |

---

## 五、动作类型层（Action Types）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **interaction_action** | 交互动作 | 需要与用户交互澄清的动作，如 `ask_clarification`（请求澄清）。通过 User/UI Channel 处理。 |
| **execution_action** | 执行动作 | 需要执行具体操作的动作，如 `probe_step`（探测）、`execute_registered_action`（执行注册动作）。经过 Guard → Runtime → Resource。 |
| **supervision_action** | 监督动作 | 需要人工监督或审批的动作，如 `request_human_confirm`（请求人工确认）。通过 Human in Loop Channel 处理。 |

## 六、观察结果层（Observations）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **Observation** | 观察结果 | 各种执行结果的统称，必须回流成结构化数据而非散乱文本。 |
| **Observation Envelope** | 观察结果外壳 | 统一的观察结果包装格式，采用"统一外壳 + 分类载荷"的设计，便于账本统一存储和 Agent 统一解析。 |
| **InteractionObservation** | 交互观察 | 用户交互的结果，包含用户回答内容。 |
| **SupervisionObservation** | 监督观察 | 人工监督的结果，包含审批结论。 |
| **ExecutionObservation** | 执行观察 | 动作执行成功的结果。 |
| **FailureObservation** | 失败观察 | 动作执行失败的结果。 |
| **BlockerObservation** | 阻断观察 | 动作被 Guard 阻断的结果。 |
| **PauseObservation** | 暂停观察 | 动作进入等待状态（如等待用户回复、等待人工审批）的结果。 |

## 七、票据工单层（Tickets）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **Interaction Ticket** | 交互工单 | 用户交互等待的工单记录，包含问题列表、响应字段等。用于支持跨会话恢复。 |
| **Approval Ticket** | 审批工单 | 人工审批等待的工单记录，包含审批 ID、响应字段等。用于支持跨会话恢复。 |
| **Resume Token** | 恢复令牌 | 恢复执行状态的标识符，用于会话中断后的恢复。 |

## 八、资源管理层（Resources）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **Resource Catalog** | 资源目录 | 注册资源动作的目录，管理"已注册动作空间"。LLM 只能在这些受控动作里选择与补参，不能直接拼任意 SQL/URL。 |
| **Resource Action** | 资源动作 | 注册的可执行资源操作，如 `order_db.insert_order`。 |
| **Resource Action Definition** | 资源动作定义 | 动作的详细定义结构，包含资源键、动作键、适配器类型、工具名、风险级别、是否支持探测、是否需要审批等。 |
| **Resource Registry** | 资源注册中心 | 动作注册和管理的中心，提供 register、get、has、list_all 等操作。 |
| **Resource Adapter** | 资源适配器 | 连接资源和工具的适配器，将受控资源动作映射到现有 xagent 工具。 |
| **SQL Adapter** | SQL 适配器 | SQL 操作适配器，把受控 SQL 动作映射到 xagent SQL 工具。不开放任意 SQL。 |
| **HTTP Adapter** | HTTP 适配器 | HTTP 操作适配器，把受控 API 动作映射到 xagent HTTP 工具。 |

## 九、执行契约层（Execution Contracts）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **Execution Contract** | 执行契约 | 标准化的执行协议，描述要执行什么、用什么参数、期望什么结果。 |
| **Compiled Execution Contract** | 编译后执行契约 | 经过 ExecutionCompiler 编译后的执行协议，可被 Runtime 稳定执行。包含 mode（probe/execute）等执行参数。 |

| **Runtime Result** | 运行时结果 | 执行结果的统一格式，包含 run_id、status、summary、data、evidence 等字段。 |

## 十、执行器层（Executors）

| 英文术语 | 中文含义 | 详细解释 |
|---------|---------|---------|
| **Runtime Executor** | 运行时执行器 | 统一执行入口，根据契约 mode 决定是走 probe_executor 还是 action_executor。 |
| **Action Executor** | 正式动作执行器 | 执行正式动作（mode=execute）的执行器。 |

## 十八、核心设计原则

### 18.1 五层架构的正确理解

```
Agent Control Plane（决策层） ← 唯一业务决策源
        ↓
Guard / Routing Plane（护栏层） ← 能阻断，不能改目标
        ↓
Runtime / Workflow Plane（执行层） ← 能执行，不能选动作
        ↓
Resource Plane（资源层） ← 能被调用，不能决定流程
        ↓
Memory / Ledger Plane（账本层） ← 贯穿全层，只存证据不指挥
```

### 18.2 关键约束

- **决策唯一源约束**：只有 Agent Control Plane 能决定下一业务动作
- **状态只读约束**：所有状态都只能作为证据输入，不能作为业务脚本
- **Guard 纯裁决约束**：可以阻断，可以给 route，不可以改目标，不可以补决策
- **Ledger 非控制器约束**：账本必须持久化，但绝不能成为隐藏的状态机脚本引擎
